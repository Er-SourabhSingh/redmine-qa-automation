BUG-RAF-001
Plugin blocks multiple dependency rules for same parent and child field combination even with different parent values
Severity: HIGH
Status: OPEN
Retest: CONFIRMED 2026-05-21
Redmine Version: 6.0.9.stable
Blocked TCs: TC-RAF-022, TC-RAF-023, TC-RAF-024, TC-RAF-036

Steps to Reproduce

  1. Navigate to Advanced Fields → Dependencies
  2. Confirm existing rule: Department = IT → Location (Server Room, Data Center, CRM) is present
  3. Fill the Add form: Parent Field = Department, Parent Value = Finance, Child Field = Location, Allowed Values = Head Office
  4. Click Add
Expected: Second dependency rule saves. Finance → Location (Head Office) appears in the rules list alongside the IT rule.
Actual: Alert fires immediately before any save attempt.
"Only one dependency rule is allowed for this Parent Field and Child Field combination."

Root Cause

The plugin enforces a uniqueness constraint per Parent Field + Child Field pair, regardless of the Parent Value. This means only a single rule can exist for any Department → Location combination, making it impossible to configure different allowed values for different department selections.

Impact

The multi-value dependency feature is effectively non-functional. Only the first configured parent value (IT) gets location filtering. All other department values (Finance, HR, Operations) see all 9 location options unfiltered.

BUG-RAF-002
Issue edit blocked by validation error on integer formula field receiving float value from client-side calculation
Severity: HIGH
Status: OPEN
Retest: CONFIRMED 2026-05-21
Redmine Version: 6.0.9.stable
Blocked TCs: None (workaround applied)

Steps to Reproduce

  1. Create an Integer-type custom field (e.g., Bug002 Repro)
  2. Configure a Divide (A ÷ B) formula: target = Bug002 Repro, A = Estimate, B = Extra Hours
  3. Create a new issue with Estimate = 10, Extra Hours = 3 — issue saves successfully; server stores 3
  4. Open the issue for edit (/issues/:id/edit)
  5. Observe: client-side JS computes 10 ÷ 3 = 3.33 and writes "3.33" into the disabled Integer input
  6. Change any non-formula field (e.g., Priority) and click Submit
Expected: Issue saves successfully. Auto-calculated fields are recalculated server-side. No validation error for disabled formula fields.
Actual: Rails validation error blocks the save.
"Bug002 repro is not a number"

Root Cause

The plugin's client-side JavaScript performs float division (10 / 3 = 3.3333...) and writes the result into the disabled Integer custom field input on page load. When the form is submitted, this float string ("3.33") is included in the POST data. Rails integer validation rejects any value that is not a whole number string, returning the validation error and blocking the edit permanently.

Workaround Applied in Testing: Changed the target custom field type from Integer to Float. Float fields accept decimal values, so Rails validation passes. All formula tests (TC-RAF-031 onwards) used Float type fields.
