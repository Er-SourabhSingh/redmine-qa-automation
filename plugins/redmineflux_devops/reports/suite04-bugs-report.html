ArgoCD provider not supported — canary and progressive rollout features entirely blocked
Affected TCsTC-RDV-174, TC-RDV-175, TC-RDV-176, TC-RDV-193, TC-RDV-194 RequirementDEP-008 — Canary/Progressive Rollout User RoleAdmin Screenshotscreenshots/TC-RDV-193/ (BLOCKED)
Steps to Reproduce
  1. POST to /projects/phoenix-platform/devops_connections with provider=argocd.
  2. Observe the HTTP response status and body.
  3. Alternatively, inspect the provider dropdown in the Add Repository form for an ArgoCD option.

BUG-RDV-037 HIGH OPEN
Multiple concurrent pending_approval deployments accepted for the same environment
Affected TCsTC-RDV-183 RequirementDEP-004 — Deployment Approval Gate User RoleAdmin Screenshotscreenshots/TC-RDV-183/ (FAIL)
Steps to Reproduce
  1. Configure environment (e.g., staging, env_id 3) with approval_required = true.
  2. POST a deploy to the environment — first deployment created with status pending_approval.
  3. Immediately POST a second deploy to the same environment without approving or rejecting the first.
  4. Check the deployments list for the environment.
Expected: Second deploy attempt is blocked with an error: "A deployment is already pending approval for this environment." Only one pending_approval deployment exists per environment at a time.
Actual: Second deploy returns HTTP 302 success. Deployments list shows multiple records with pending_approval status for the same environment simultaneously (confirmed: deployments #17, #18, #19 all showing staging/pending). Approval queue integrity is broken.
BUG-RDV-032 MEDIUM OPEN
Deploy button not disabled when environment is locked or has an active freeze window
Affected TCsTC-RDV-169, TC-RDV-181 RequirementDEP-006 / DEP-009 — Lock & Freeze pre-flight User RoleAdmin Screenshotscreenshots/TC-RDV-181/tc-rdv-181-pass.png
Steps to Reproduce
  1. Lock an environment with a future locked_until date. Navigate to /devops_environments.
  2. Observe the Deploy button on the locked environment card.
  3. Alternatively, create an active freeze window for the environment and repeat.
Expected: Deploy button is visually disabled (grayed out, disabled attribute, or hidden) with a tooltip indicating the environment is locked or frozen.
Actual: Deploy button remains fully enabled for locked and frozen environments. Server correctly enforces the block (HTTP 423 for freeze, flash "Deployment blocked — environment is locked" for lock), but the UI gives no pre-emptive indication. Users discover the block only after attempting deployment.

Steps to Reproduce
  1. Send a GitHub deployment_status webhook with a valid deployment.sha and deployment_status.target_url (e.g., TC-RDV-195 webhook payload).

BUG-RDV-036 MEDIUM OPEN
After rollback, original deployment detail shows no "Reverted by" reverse link
Affected TCsTC-RDV-199 RequirementDEP-005 — Rollback tracking / bi-directional audit trail User RoleAdmin Screenshotscreenshots/TC-RDV-176/
Steps to Reproduce
  1. Navigate to a successful deployment (e.g., deployment #5).
  2. Create a rollback from that deployment via /devops_deployments/5/create_rollback.
  3. Confirm the rollback deployment is created (new record visible in list).
  4. Navigate back to the original deployment detail page (#5).
Expected: Original deployment detail (#5) shows a "Reverted by deployment #N" reference with a link to the rollback record, completing the bi-directional audit trail.
Actual: Original deployment #5 shows no "Reverted by" section. The rollback detail correctly links back to the original ("Rollback of #5"), but the reverse link is absent — tracing from original to rollback is not possible via UI.
